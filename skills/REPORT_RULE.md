## Technical Documentation Rule

### 1. Documentation Process (Trigger)

- **🐛 [fix] / 🚑 [hotfix] 커밋 발생 시**: 해당 트러블슈팅 내역을 즉시 `TROUBLE_SHOOTING.md` 파일에 아래 양식으로 기록합니다.
- **주요 architectural 변경 발생 시**: 비즈니스 로직과 무관한 시스템 설계상의 의사결정 과정을 `Why/What/How/Result` 형식으로 기록합니다.

### 2. Portfolio Storytelling Structure

문제 정의 → 가설 설정 → 가설 검증 및 해결 → 정량적 결과.
**문체는 반드시 존댓말(~했습니다, ~입니다)로 작성한다. 포트폴리오는 면접관이 읽는 문서이기 때문이다.**
**비즈니스 로직 소스 코드는 포함하지 않는다. 트러블슈팅 해결 과정의 코드/커맨드는 포함 가능.**

### 3. Why / What / How / Result format

헤딩 레벨과 넘버링 양식은 아래를 엄격히 따른다:

```
1. [문제/주제 제목]

###### Why: 무엇을 해결하고 싶었나요?
- 문제 정의: 왜 이 문제가 발생했는지, 어떤 증상/병목이 있었는지.
- 위험 요소: 해결하지 않으면 시스템/비즈니스에 어떤 리스크가 있는지.

###### What: 그래서 무엇을 만들었나요?
- 가설 설정: 어떤 접근을 선택했는지, 대안 대비 왜 이 방법인지.
- 기술적 근거: 선택한 해결책의 기술적 타당성 (공식 문서 등 근거 제시).

###### How: 이 목표를 어떻게 해결했나요?
1) [첫 번째 해결 단계 제목]: 가설을 검증하고 적용한 구체적 과정.
2) [두 번째 해결 단계 제목]: ...

###### Result: 어떤 결과를 얻었나요?
- 정량적 수치: 이 해결책이 가져다준 지표 (ms, %, 배수 등).
- 회고 및 예방: 재발 방지를 위해 어떤 시스템적 조치(Lint, Test, Rule)를 취했는가.
```

### 4. Troubleshooting log format

**트러블슈팅 과정을 마쳤으면 반드시 포트폴리오 문서(Why/What/How/Result 형식)에 기록한다.**
각 트러블슈팅 항목은 반드시 아래 4개 필드 + 추가 정보를 포함한다:

- **증상**: 관찰된 현상 + 에러 메시지 원문 포함 가능 (비즈니스 로직 무관)
- **원인**: 근본 원인 (Root Cause)
- **해결**: 적용한 해결 방법 — 코드/커맨드 포함 가능 (트러블슈팅 포트폴리오 목적에 부합)
- **재현 조건**: 언제 다시 발생할 수 있는지.
- **참조**: 해결 과정에서 참고한 공식 문서(`context7` 검색 결과 등)의 링크 또는 핵심 키워드.
